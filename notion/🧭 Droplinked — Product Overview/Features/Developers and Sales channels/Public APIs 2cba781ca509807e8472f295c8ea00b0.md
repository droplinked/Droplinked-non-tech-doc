# **1) Summary**

### **Problem / Value**

Merchants and developers want to build custom storefronts, headless applications, or integrate Droplinked into existing platforms. To achieve this, they need programmatic access to shop data (Products, Carts, Orders) and clear instructions on how to use it.
This feature exposes the **Public API endpoints** and provides a direct gateway to the technical documentation (GitBook). It relies on the **API Keys** generated in the previous step to authenticate and secure these requests.

### **Desired Outcome**

- Expose public API endpoints for core e-commerce functions (Product, Cart, Checkout, etc.).
- Secure all endpoints using the API Keys generated in the **Developers** settings.
- Provide a direct link to the **API Reference (GitBook)** within the Merchant Dashboard so developers can easily learn how to structure requests.
- Ensure strict authentication: Requests without a valid key must be rejected.

---

# **2) Scope – In / Out**

### **In Scope**

- **Documentation Access:**
    - Display a "View API Documentation" link in the **Settings > Developers** tab (near the API Key management section).
    - Link destination: `https://droplinked.gitbook.io/droplinked-store-front-help-center/library/droplinked-api/api-reference`.
- **API Security:**
    - Authentication via Header (e.g., `x-api-key` or Bearer Token).
    - Validation of the API Key against the shop's active key record.
- **Endpoint Availability:**
    - Access to Product data (Read).
    - Access to Cart/Checkout flows (Read/Write).
- **Error Handling:** Return standard HTTP error codes (401 Unauthorized, 403 Forbidden) for invalid or expired keys.

### **Out of Scope**

- Generating the API Key [(Handled in **DEV-API-KEY-011**)](../Shop%20builder/Shop%20Settings/API%20Keys%20Management%202cba781ca50980d5aa59cc47a92b79e6.md).
- SDK generation (client libraries).
- Webhook management (separate feature).

# **3) Key User Journeys**

---

### **Journey 1 – Accessing Documentation**

**Step 1:** Merchant/Developer navigates to **Settings > Developers**.
**Step 2:** They see the section for API Keys (created in previous feature).
**Step 3:** They click the link **"Read API Documentation"** or **"View Docs"**.
**Step 4:** System opens the **Droplinked GitBook** reference in a new tab.

---

### **Journey 2 – Making a Valid API Request**

**Pre-condition:** Merchant has generated an API Key via **DEV-API-KEY-011**.
**Step 1:** Developer reads the GitBook to find the "Get Products" endpoint.
**Step 2:** Developer sends a GET request to the endpoint, including the **API Key** in the request header.
**Step 3:** System validates the key is Active and belongs to the Shop.
**Step 4:** System returns the requested JSON data (200 OK).

[https://www.notion.so/API-Keys-Management-2cba781ca50980d5aa59cc47a92b79e6](../Shop%20builder/Shop%20Settings/API%20Keys%20Management%202cba781ca50980d5aa59cc47a92b79e6.md)

---

### **Journey 3 – Invalid/Missing Key**

**Step 1:** Developer attempts to call an API endpoint without a key or with a **Revoked/Expired** key.
**Step 2:** System rejects the request.
**Step 3:** System returns **401 Unauthorized** error message.

---

# **4) Business Acceptance Criteria (BAC)**

**BAC 1:** A visible link to the [Official API Documentation](https://droplinked.gitbook.io/droplinked-store-front-help-center/library/droplinked-api/api-reference) must be present in the Developers Settings tab.
**BAC 2:** Public API endpoints must **require** a valid API Key to function.
**BAC 3:** Requests made with a valid, active API Key must return the correct shop data.
**BAC 4:** Requests made with a **Revoked**, **Expired**, or **Non-existent** key must return a **401 Unauthorized** response immediately.
**BAC 5:** The API must support cross-origin requests (CORS) to allow usage in headless web applications (if applicable).
